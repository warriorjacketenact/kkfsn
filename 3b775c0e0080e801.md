# TekStack Resource Gateway

TekStack Resource Gateway 是一个面向技术团队与基础设施管理者的外链资源汇总与导航系统，专注于将分散于多个代码仓库、文档站点与内部知识库中的关键外部链接统一收录、分类归档与版本化跟踪。该项目本身不托管文档内容，而是提供结构化索引，帮助组织在微服务架构演进过程中保持外部依赖的可追溯性。

项目定位为技术资源治理工具，目标用户包括架构师、运维工程师、技术文档负责人以及新入职的研发人员。通过集中管理外链资源，TekStack Resource Gateway 能够解决因文档散乱导致的链接失效、上下文缺失、资源重复引用等问题，同时为技术决策提供可量化的外部依赖视图。

## 功能概览

**批量资源导入** 支持从 Markdown 文档、CSV 文件与 JSON 清单中批量解析外部链接，自动去重并生成统一资源标识符。

**分类标签系统** 允许用户为每个资源记录自定义标签与分类层级，支持多维度筛选与组合查询。

**链接状态监测** 定期对已收录的 URL 发起 HTTP 请求，检测响应状态码与重定向链，标记失效或变更的资源。

**版本快照对比** 记录每次资源列表的更新历史，支持任意两个版本之间的差异对比，显示新增、删除与修改的条目。

**Markdown 原生渲染** 所有资源清单以纯 Markdown 格式输出，可直接嵌入项目 README、技术文档或静态站点生成器。

**权限分级管理** 内置基于角色的访问控制，区分管理员、编辑者与只读用户，适用于企业级内部部署。

**自定义元数据扩展** 每条资源可附加键值对形式的自定义字段，如负责人、所属项目、关联工单编号等。

**全文检索与过滤** 提供基于标题、描述、标签与资源路径的全文搜索，支持正则表达式过滤与批量操作。

## 应用场景

企业内部文档集中治理。当技术团队同时维护数十个微服务仓库时，每个仓库的 README 可能重复引用相同的第三方库文档或内部设计文档。TekStack Resource Gateway 可作为统一入口，将所有外链收录至单一索引文件中，各仓库仅需引用该索引的锚点，减少重复维护成本。

技术栈升级影响评估。在进行框架版本升级或迁移底层基础设施时，管理员可通过本项目的版本快照对比功能，快速定位所有引用旧版本文档或已弃用 API 的资源，从而评估升级影响范围并制定迁移计划。

新员工入职学习路径构建。将团队常用的技术博客、官方手册、内部培训视频链接按学习阶段分类打包，生成结构化的学习资源清单。新员工可以通过该清单按顺序访问各类资料，避免信息过载与盲目搜索。

外部依赖合规审计。安全合规团队需要定期审查项目对外部服务的依赖情况。本系统导出的资源清单可与审计流程对接，提供完整的依赖 URL 列表及其变更历史，满足内部合规与外部监管要求。

## 快速开始

以下操作假设您已安装 Git 与 Node.js 运行时环境。

```bash
git clone https://github.com/tekstack/resource-gateway.git
cd resource-gateway
npm install
npm run build
npm start -- --input ./samples/import.json --output ./dist/resource-index.md
```

首次启动时，系统会在当前目录生成默认配置文件 `gateway.config.js`，您可以根据需要调整资源存储路径、监测间隔与端口号。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 18.x 或 20.x LTS | 运行时环境，建议使用官方二进制或 NVM 管理 |
| npm | 9.x 或 10.x | 包管理器，用于安装项目依赖 |
| Git | 2.30 及以上 | 版本控制，用于克隆仓库与提交更新 |
| SQLite3 | 系统自带或通过 npm 安装 | 轻量级嵌入式数据库，存储资源索引与变更历史 |
| curl 或 wget | 系统自带 | 用于链接状态监测模块的 HTTP 探测 |
| OpenSSL | 1.1.1 或 3.x | 用于生成签名与加密敏感配置项 |
| 磁盘空间 | 至少 200 MB | 存储数据库文件、日志与临时缓存 |
| 内存 | 最低 512 MB，推荐 1 GB | 运行服务与执行批量扫描任务 |

## 贡献指南

我们欢迎并鼓励社区贡献。请按照以下步骤提交您的改进。

第一，查阅问题列表与路线图。访问 GitHub Issues 查看当前已知问题与待实现功能，确认您要解决的任务未被他人认领。对于新功能提议，请先创建 Issue 进行需求讨论，避免无效开发。

第二，派生仓库并创建功能分支。将主仓库派生至您的个人账户，然后克隆至本地。创建以 `feature/` 或 `fix/` 为前缀的分支，例如 `feature/add-json-export`。确保分支名称简明描述变更内容。

第三，编写代码与测试用例。所有新增功能必须包含对应的单元测试，测试覆盖率不得低于 80%。遵循项目已有的代码风格与 ESLint 规则，提交前执行 `npm run lint` 与 `npm test` 确保全部检查通过。

第四，提交变更并创建拉取请求。提交信息应采用语义化格式，首行概括变更内容，正文详细说明动机与实现方式。推送分支后，在主仓库创建拉取请求，并填写 PR 模板中的检查清单。核心维护者将在三个工作日内进行评审。

第五，签署开发者原创声明。首次贡献者需在 PR 评论中明确声明代码为本人原创，且同意按照 MIT 许可证授权。该声明将作为合并的前置条件。

## 常见问题

**问：资源列表中的链接发生变更或失效，系统如何处理？**

答：监测模块会根据配置的周期（默认每 24 小时）对所有已收录的 URL 发起 HEAD 或 GET 请求。若返回状态码为 4xx 或 5xx，系统会在日志中记录该事件，并通过配置的通知渠道（邮件、Webhook 或 Slack）向管理员发送告警。管理员可手动标记该资源为“已失效”或更新其 URL。此外，版本快照功能保留历史记录，便于追溯链接变更的时间点。

**问：是否支持从私有仓库或需要认证的源导入资源？**

答：支持。您可以在配置文件中为特定来源设置请求头或基本认证凭据。导入模块会在发起请求时自动附加这些认证信息。对于需要 OAuth 或令牌认证的源，建议通过环境变量注入凭证，避免将敏感信息硬编码至配置文件中。详细配置方式请参考部署运维指南中的安全章节。

**问：如何将导出的 Markdown 资源清单集成到现有的文档站点中？**

答：导出命令默认生成纯 Markdown 文件，该文件可直接被 MkDocs、VuePress、Docusaurus 等静态站点生成器引用。您可以将导出的文件放置在站点的文档目录下，并在导航栏中添加对应的链接项。如果希望保持实时同步，可以配置 CI 流水线在每次资源更新后自动执行导出并提交至文档仓库。
