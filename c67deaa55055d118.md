# Tekton Resource Aggregator

Tekton Resource Aggregator 是一个面向技术文档工程师、开源贡献者以及开发者关系维护者的结构化外链与资源汇总工具。该项目不存储任何实际内容，仅以纯 Markdown 形式提供经过分类与注释的高质量外部资源索引，旨在解决个人收藏夹无序、团队共享困难以及技术调研重复劳动等问题。

项目定位为轻量级静态资源导航，所有条目均以哈希命名文件形式存储在仓库主干中，便于自动化脚本批量处理与版本追踪。目标用户包括需要维护技术周报的编辑、进行竞品分析的产品经理、以及搭建内部技术知识库的架构师团队。

## 功能概览

**批量链接导入** 支持通过脚本将散落的 URL 列表一键转换为标准格式的 Markdown 条目，自动生成哈希文件名并归类至对应目录。

**哈希索引管理** 每个资源条目以唯一哈希值命名，避免文件名冲突，同时保留原始 URL 与添加时间戳的元数据记录。

**分类标签体系** 内置技术栈、语言、领域、来源类型四维标签，允许用户通过正则表达式或 grep 命令快速筛选特定类别资源。

**只读镜像部署** 项目本身不依赖数据库或后端服务，可直接 fork 后通过静态托管服务（如 GitHub Pages）生成只读资源目录页。

**版本差异对比** 利用 Git 原生 diff 能力，追踪每个资源条目的增删改历史，便于团队审计外部链接的变更记录。

**自动化校验脚本** 提供 Python 辅助脚本，周期性检查已收录链接的可达性，标记失效链接并生成报告。

**自定义模板生成** 用户可通过修改模板文件，自定义每个资源条目的输出字段，如摘要、优先级、过期时间等。

## 应用场景

技术团队内部知识库建设
团队可将本仓库作为外部资源的中转站，成员通过提交 PR 的方式新增有用链接，经审核合并后自动更新团队共享的资源池，避免重复搜索与信息孤岛。

开源项目文档外链管理
开源项目维护者可以使用本工具集中管理 README、官网、教程等各类外链，当外部站点迁移或改版时，只需在索引文件中更新一条记录即可同步至所有引用处。

技术调研与竞品监控
产品与技术调研人员可定期批量导入竞品发布公告、版本日志、社区讨论等链接，利用哈希版本控制追踪竞品动态的时间线，形成结构化调研档案。

个人开发环境书签同步
开发者可将个人分散在浏览器、笔记软件、即时通讯中的技术书签统一导出为 Markdown 列表，通过本项目的导入脚本完成集中归档，并在多设备间通过 Git 同步。

## 快速开始

以下命令演示了从克隆仓库到运行本地预览服务的完整流程。

```bash
# 克隆仓库到本地
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg

# 安装 Python 依赖（用于辅助脚本）
pip install -r requirements.txt

# 运行本地静态站点生成（若有配置）
python scripts/build_index.py --input ./main --output ./dist
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Git | 是 | 用于克隆仓库、版本控制与提交变更 |
| Python 3.8+ | 是 | 运行链接校验、索引构建等辅助脚本 |
| pip | 是 | 安装 Python 依赖包 |
| Markdown 解析器（如 mistune） | 否 | 仅当需要本地渲染预览时需要 |
| 静态托管服务（如 Nginx） | 否 | 生产环境部署静态页面时使用 |
| cron 或 systemd timer | 否 | 用于定时执行链接可达性校验任务 |

## 贡献指南

提交新资源条目
贡献者通过 fork 仓库，在 main 目录下新建一个以哈希命名的 .md 文件，内容须包含完整 URL、分类标签以及简短的摘要说明。提交 PR 后由维护者审核合并。

更新已有资源
若需修改某条目的 URL 或标签，请直接编辑对应的哈希文件，并在 PR 描述中注明变更原因。禁止重命名或删除已有文件，以保证版本历史可追溯。

运行校验脚本
在提交前，建议本地执行 python scripts/check_links.py 以验证所有外部链接的有效性。如发现失效链接，请及时更新或标记为过期。

编写或更新文档
文档目录下的所有手册与指南欢迎改进。请遵循现有的 Markdown 风格，保持章节层级清晰，并确保代码示例可执行。

报告问题与建议
使用 GitHub Issues 提交 bug 报告或功能请求。请附上重现步骤、日志截图以及预期行为描述，便于维护者快速定位。

## 常见问题

问：为什么所有条目都用哈希值命名，而不是用可读的英文名？

答：哈希命名可以避免文件名中的特殊字符、空格及长度限制问题，同时保证每个条目在 Git 历史中拥有唯一的标识符，便于脚本自动化处理。当需要查找具体条目时，可通过 grep 搜索哈希文件内的原始 URL 或标签来定位。

问：如何处理失效的外部链接？

答：项目提供了 check_links.py 脚本，可定期扫描所有条目的 URL 并返回 HTTP 状态码。对于返回 4xx 或 5xx 的链接，建议人工访问确认是否永久迁移或彻底关闭。若为永久迁移，更新对应哈希文件中的 URL 字段；若已关闭，则标记 status 字段为 deprecated。

问：如何将已有的浏览器书签批量导入？

答：大多数浏览器支持将书签导出为 HTML 文件。用户可先将该 HTML 文件中的链接提取为纯文本列表，每行一个 URL，然后使用 scripts/import_batch.py --input bookmarks.txt 命令一次性生成对应的哈希条目文件。导入后建议手动补充摘要与标签。
