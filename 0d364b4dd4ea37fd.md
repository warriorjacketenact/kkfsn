# ResourceIndex Core

ResourceIndex Core 是一个面向技术文档聚合与外部资源索引的开源基础设施项目。项目定位为技术资源导航与结构化外链汇总系统，主要服务于需要批量管理、分类展示和快速检索大量外部技术链接的开发者、技术内容运营团队以及知识管理研究人员。系统核心目标在于解决分散资源难以组织、链接失效难以追踪、分类维度混乱等实际问题，提供基于 Markdown 与 Git 工作流的轻量级资源编排方案。

本项目当前批次为第 30/76 批，累计已收录并维护 180 个外部技术资源链接。通过标准化条目格式与目录树索引机制，ResourceIndex Core 能够将松散的外部 URL 转化为具有可读性、可维护性和可扩展性的结构化文档集合。

## 功能概览

**批量链接导入解析**：支持从纯文本列表、CSV 以及 Markdown 表格中批量提取 URL，自动识别协议头与路径结构，并生成初始条目模板。

**多维度标签分类**：每个资源条目可赋予技术领域、内容类型、难度等级、适用场景等多重标签，支持按任意标签组合进行筛选与视图切换。

**链接状态健康检查**：集成基于 HTTP 状态码的链接可用性探测，自动标记失效或重定向链接，并生成异常报告。

**全文检索与模糊匹配**：基于标题、描述、标签及文件路径构建轻量级倒排索引，支持中英文关键词快速定位目标资源。

**目录树自动生成**：根据资源分类层级与文件存储结构，动态生成 ASCII 目录树视图，直观展示项目组织方式。

**变更历史审计追踪**：基于 Git 提交记录自动关联资源增删改操作，提供按时间、作者、操作类型的审计日志查询。

**模板化文档输出**：内置 README、分类索引、场景导航等多种文档模板，可一键生成面向不同受众的展示页面。

**增量更新与合并策略**：支持多批次资源增量导入，自动去重并提示冲突条目，提供交互式与自动化两种合并模式。

## 应用场景

技术团队内部知识库建设：技术负责人可使用 ResourceIndex Core 整理团队积累的 API 文档、运维手册、架构设计稿等外部参考链接，按项目或业务域分类，形成统一的知识入口，降低新成员上手的信息检索成本。

开源项目外链文档维护：开源项目维护者可将项目依赖的规范、教程、工具站点、社区论坛等外链资源纳入统一索引，随项目版本发布同步更新，确保文档中的外部引用始终可访问且版本匹配。

技术资讯与论文汇总管理：研究人员或技术编辑可批量收录 arXiv 论文、技术博客、会议录播链接，按主题、时间、会议等级等多维度组织，配合健康检查功能定期清理失效链接，保证资源集的有效性。

个人技术收藏夹系统化改造：开发者可将浏览器收藏夹中散落的技术链接导出后导入 ResourceIndex Core，通过标签和分类重新整理，并利用全文检索快速找回特定主题下的收藏内容，替代传统书签管理的扁平化局限。

## 快速开始

以下命令演示如何从 GitHub 克隆项目、安装依赖并启动本地预览服务。

```bash
git clone https://github.com/your-username/resourceindex-core.git
cd resourceindex-core
pip install -r requirements.txt
python scripts/serve.py --port 8080
```

执行上述命令后，可通过浏览器访问本地 8080 端口查看资源索引首页。如需导入用户提供的原始链接数据，请将链接列表保存为纯文本文件（每行一个 URL）并放置于 `data/raw/` 目录下，然后执行导入命令：

```bash
python scripts/import.py --source data/raw/links.txt --batch 30
```

## 贡献指南

1.  Fork 本仓库至个人账户，并在本地克隆 Fork 后的副本。创建以 `feature/` 或 `fix/` 为前缀的新分支，用于隔离开发改动。

2.  在 `data/raw/` 目录下按批次格式添加新链接列表文件，或修改 `src/` 下的核心逻辑时，请同步更新 `docs/` 中对应的文档说明，确保功能变更与文档描述一致。

3.  执行 `pytest tests/` 运行全部测试套件，确保新增代码未破坏现有功能。若新增功能模块，请在 `tests/` 下补充对应的测试文件。

4.  提交代码前使用 `black .` 格式化全部 Python 源文件，保持代码风格统一。提交信息遵循约定式提交规范，格式为 `<类型>: <简短描述>`，例如 `feat: add batch merge strategy`。

5.  发起 Pull Request 到主仓库的 `main` 分支，在 PR 描述中清晰说明改动目的、影响范围以及关联的批次编号或 Issue 编号。PR 至少需要一名维护者审核通过后方可合并。

## 常见问题

**问：导入链接时提示哈希冲突，该如何处理？**

导入器会根据 URL 字符串计算 SHA-256 哈希作为条目文件名。如果两个不同的 URL 产生相同哈希（极低概率的碰撞情况），或同一 URL 在批次内重复出现，系统会输出冲突警告并跳过重复条目。解决方法是检查原始数据是否存在复制粘贴错误，或手动编辑 `data/parsed/entries/` 下的冲突文件，使用带时间戳的后缀重命名。对于批次内重复，可在导入命令中添加 `--dedup` 参数自动去重。

**问：链接健康检查报告大量超时或 SSL 错误，是否意味着所有链接均失效？**

不一定。健康检查模块默认超时时间为 10 秒，对于响应较慢的服务器或网络环境不稳定时可能出现误报。此外，部分站点禁用了 HEAD 请求或存在 SSL 证书链问题。建议先使用 `--timeout 30` 参数延长等待时间，并启用 `--verify-ssl false` 忽略证书校验后重新检查。若仍大面积报错，可能是目标站点集体屏蔽了自动化请求，可考虑调整 User-Agent 或增加请求间隔。

**问：如何在本地预览时只显示特定标签下的资源？**

启动本地服务后，访问 `http://localhost:8080/?tag=<标签名>` 即可按标签过滤。也支持多标签组合，使用逗号分隔，例如 `?tag=python,kubernetes`。若需在命令行下导出过滤结果，可使用 `scripts/build_index.py --filter tag=python --output filtered.json` 生成 JSON 格式的筛选列表。
