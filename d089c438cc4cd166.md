# CSU-TKG Resource Index

CSU-TKG Resource Index 是一个面向技术研究与开发人员的结构化外链资源汇总系统。该项目以 GitHub 仓库为载体，通过十六进制哈希命名规范组织大量技术文档、教程、规范及工具链接，旨在为开发者提供一套可检索、可回溯、可版本化的外部知识索引体系。项目本身不托管原始内容，而是维护一个严谨的引用映射表，确保每一条外部资源都能被稳定定位和快速访问。

该项目主要服务于需要频繁查阅分散技术资料的全栈工程师、运维人员、技术作者以及知识管理爱好者。通过将散落在互联网各处的优质内容以 Markdown 文件的形式建立本地索引，用户可以在离线环境下预览资源摘要，或通过脚本批量检查链接可用性，从而有效避免书签泛滥和链接失效的问题。

## 功能概览

**结构化索引存储**：所有资源以 SHA-1 哈希值作为文件名，统一存放于仓库 main 分支的根目录下，确保命名唯一且具备内容寻址特性。

**Markdown 元数据模板**：每个索引文件内部包含资源标题、原始 URL、收录日期、分类标签及摘要说明，便于后续解析和展示。

**批量链接状态检查**：项目提供配套的 Python 脚本，可定期对所有收录的 URL 进行 HTTP 状态码检测，自动标记失效链接。

**分类标签体系**：内置多级标签系统，涵盖编程语言、框架、算法、运维、数据库等常见技术领域，支持快速筛选。

**版本化变更记录**：每次新增或更新索引文件均通过 Git 提交完成，保留完整的增删改历史，便于追溯资源变动。

**全文检索支持**：借助 GitHub 原生搜索功能，用户可对索引文件内容进行关键字检索，快速定位相关资源。

**自动生成目录树**：通过 CI 脚本自动生成项目结构树状图，并嵌入 README，方便用户直观了解文件分布。

## 应用场景

**技术文档归档**：技术团队可将日常开发中参考的官方文档、博客文章、API 手册等外链统一收录至项目仓库，形成团队共享的知识索引库，新人入职时可快速拉取并浏览全部推荐资源。

**在线课程辅助学习**：自学者在跟随在线课程或技术书籍时，可将每章节涉及的扩展阅读链接录入系统，配合 Git 提交记录跟踪学习进度，避免因浏览器书签丢失而中断学习计划。

**开源项目外部依赖整理**：开源项目维护者可将项目依赖的第三方工具、库文档、配置文件示例等外部资源链接集中管理，在项目 README 中引用本仓库作为“外部资源附录”，降低新贡献者的上手门槛。

**技术资讯聚合**：技术博主或资讯编辑可利用本项目的分类标签体系，将不同领域的新闻源、周报链接、趋势分析文章分门别类存储，每周批量导出生成资讯汇总邮件。

## 快速开始

以下命令演示了如何将本项目克隆至本地、安装基础依赖并运行链接检查脚本。

```bash
# 克隆仓库到本地
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg

# 安装 Python 依赖（建议使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install requests beautifulsoup4

# 运行链接状态检查脚本（仅检查最近新增的 50 条）
python scripts/check_links.py --limit 50
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Git | 是 | 用于克隆仓库及提交变更，版本不低于 2.25.0 |
| Python 3.8+ | 是 | 运行辅助脚本及链接检查工具的核心解释器 |
| pip | 是 | 安装 Python 依赖包管理器，版本不低于 20.0 |
| requests 库 | 是 | 用于发送 HTTP 请求检测链接状态，版本 2.28.0+ |
| beautifulsoup4 库 | 否 | 可选依赖，用于解析 HTML 提取页面标题，提升元数据质量 |
| markdown 库 | 否 | 可选依赖，用于本地渲染 README 预览，版本 3.4.0+ |
| pytest 库 | 否 | 仅开发测试时使用，用于运行单元测试用例 |

## 项目结构

项目根目录下的文件和文件夹组织如下，每个主要目录均承担特定的功能职责。

```
csutkg/
├── .github/                         # GitHub 自动化工作流配置目录
│   └── workflows/                   # CI 脚本，用于定时检查链接和更新目录树
├── scripts/                         # 可执行工具脚本存放目录
│   ├── check_links.py               # 遍历所有索引文件并检测 URL 状态码
│   ├── generate_tree.py             # 自动生成项目目录树并写入 README
│   └── import_batch.py              # 批量导入新链接并生成哈希文件名
├── docs/                            # 项目文档目录，包含详细使用手册
│   ├── quickstart.md                # 面向新用户的快速上手指南
│   ├── specification.md             # 索引文件格式规范与字段定义
│   ├── scripts.md                   # 各脚本的详细参数说明与示例
│   └── maintenance.md               # 日常维护流程与版本发布策略
├── tests/                           # 单元测试目录，用于验证脚本逻辑
│   ├── test_check_links.py          # 链接检查函数的测试用例
│   └── test_import_batch.py         # 批量导入功能的测试用例
├── templates/                       # 索引文件模板目录
│   └── index_template.md            # 新建索引时复制的基础模板
├── .gitignore                       # Git 忽略文件列表，排除虚拟环境和缓存
├── LICENSE                          # MIT 许可证全文
├── README.md                        # 项目主文档（即当前文件）
└── requirements.txt                 # Python 依赖声明文件，供 pip 安装使用
```

## 贡献指南

我们欢迎社区贡献者以多种方式参与本项目，包括新增资源链接、修复失效 URL、改进脚本效率或完善文档。请遵循以下步骤提交贡献。

第一，Fork 本仓库至个人账户，并克隆到本地开发环境。请确保本地 Git 配置正确，且已安装 Python 3.8 及以上版本。

第二，在新增索引文件时，务必从 templates/index_template.md 复制模板，并按照 docs/specification.md 中定义的字段规范填写标题、原始 URL、分类标签及摘要。文件名必须使用 SHA-1 哈希值，该值由原始 URL 计算得出。

第三，所有脚本改动需附带对应的单元测试，并确保现有测试用例全部通过。测试执行命令为 pytest tests/，请在提交前运行完整测试套件。

第四，提交变更时使用清晰的 commit message，格式为 <类型>: <简短描述>，例如 feat: 新增 30 条 Kubernetes 相关资源索引 或 fix: 更新 5 条失效的文档链接。

第五，发起 Pull Request 至主仓库的 main 分支，并在 PR 描述中说明变更内容、影响范围及测试结果。PR 通过 CI 检查且获得至少一位维护者批准后即可合并。

## 常见问题

问：索引文件的哈希文件名是如何生成的？

答：每条索引文件名的 16 进制字符串（如 438b2b687c02adf9）是原始 URL 经过 SHA-1 哈希运算后取前 16 位得到的。该机制确保同一 URL 始终映射到相同文件名，避免重复收录。用户无需手动计算，运行 scripts/import_batch.py 脚本时会自动完成哈希生成和文件命名。

问：如何批量检查所有收录链接的可用性？

答：在项目根目录下执行 python scripts/check_links.py 即可遍历所有 .md 文件，提取其中的 URL 并发送 HEAD 请求检测状态码。脚本默认输出失效链接列表及对应的 HTTP 状态码，支持 --timeout 参数调整超时时间，也支持 --output 参数将结果导出为 CSV 文件。

问：如果原始资源链接发生变更，应该如何更新索引？

答：当发现某个索引文件指向的原始内容已迁移至新 URL 时，请直接编辑对应的 .md 文件，修改其中的原始 URL 字段，并更新摘要说明中关于链接变更的备注。随后提交变更并附带 fix: 前缀的 commit message。脚本不会自动重命名文件，因为文件名基于旧 URL 的哈希，若新旧 URL 不同则建议新建索引文件并弃用旧文件。
