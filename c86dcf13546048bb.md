# CSUtkg Resource Index

CSUtkg Resource Index 是一个面向技术研究与开发人员的结构化外部资源导航项目。本项目不托管具体内容，而是对分散在多个数据仓库、技术博客、官方文档与社区讨论中的高质量外部链接进行系统性整理与分类索引，帮助用户在特定技术栈或问题域中快速定位权威参考资料。

项目定位为"技术资源的外链汇总与语义索引站"，主要服务于以下三类用户：需要快速查找某类开源组件实践案例的研发工程师、进行技术选型或竞品分析的系统架构师、以及希望跟进特定领域最新动态的研究人员。通过统一的条目标识与分类体系，用户无需记忆散乱的网址或文档片段，即可通过本项目提供的索引树与分类导航直达目标资源。

## 功能概览

- **按技术领域分类索引**：将资源按照编程语言、框架、中间件、云原生、数据工程等维度进行一级分类，每个分类下进一步拆分二级子主题，便于用户按领域浏览。

- **按资源类型筛选**：支持按文档类型（官方手册、社区博文、视频教程、源码仓库、技术白皮书）进行筛选，满足不同阅读习惯与使用场景。

- **全文检索与模糊匹配**：基于资源标题、摘要、关键词与标签构建轻量级检索能力，支持中英文混合查询，帮助用户在海量外链中精准定位。

- **版本与时效性标注**：对每个外链资源标注发布或更新日期、对应软件版本号以及时效性状态（稳定、已废弃、待更新），辅助用户判断内容的可用性。

- **外部链接可用性探测**：定期对已收录的 URL 发起 HEAD 请求，检测链接是否可访问，并在索引中标注异常状态，降低用户遇到死链的概率。

- **用户自定义标签与收藏**：允许用户在本地副本或私有 fork 中为资源添加自定义标签与收藏标记，便于个人知识管理。

- **批量导入与导出**：支持通过 CSV 或 JSON 格式批量导入外部链接列表，也支持将当前索引数据导出为结构化文件，方便迁移或二次处理。

- **Markdown 文档自动生成**：项目内置模板引擎，可根据索引数据自动生成符合团队规范的 Markdown 文档，用于内部周报、技术雷达或知识库归档。

## 应用场景

**场景一：新技术选型时的资料搜集**
团队计划引入消息队列 Kafka 替代原有系统，架构师需要对比 Kafka 与 RabbitMQ 的性能表现、部署模式与生态工具。通过本项目"消息中间件"分类下的索引，可快速找到 Kafka 官方性能测试报告、社区最佳实践案例以及多种部署方案对比文章，显著缩短调研周期。

**场景二：故障排查时的知识回溯**
线上服务出现内存泄漏，运维工程师怀疑与某个第三方库的特定版本有关。通过本项目的版本标注与标签筛选功能，可以快速定位该库对应版本的已知问题列表、社区讨论线程以及临时规避方案，帮助团队在短时间内定位根因。

**场景三：技术培训与新人 onboarding**
新入职的开发人员需要系统学习公司技术栈中涉及的各类组件。培训负责人可基于本项目整理一份"新手必读"资源列表，涵盖语言基础、框架入门、工具链配置以及内部规范文档，形成标准化的学习路径，降低培训成本。

## 快速开始

以下命令演示了如何将本项目克隆到本地、安装依赖并启动索引服务。

```bash
git clone https://github.com/your-org/csutkg-resource-index.git
cd csutkg-resource-index
npm install
npm start
```

执行完成后，本地服务默认运行在 3000 端口，用户可通过浏览器访问 http://localhost:3000 查看索引首页。如需导入示例数据，可执行：

```bash
npm run seed
```

该命令将载入预设的分类结构与资源条目，供试用与评估。

## 贡献指南

欢迎并感谢所有形式的贡献。请遵循以下步骤参与本项目开发。

第一步：阅读贡献规范文档
在提交任何内容之前，请先阅读 docs/guides/contribution-style.md 文档，了解资源条目的格式要求、分类标签的使用约定以及描述文本的撰写风格。不符合规范的提交将被要求修改。

第二步：Fork 仓库并创建特性分支
从主仓库 fork 一份副本到个人账号下，然后在本地基于 main 分支创建新的特性分支，分支命名建议采用 feat/功能简述 或 fix/问题简述 格式，例如 feat/add-kafka-best-practice。

第三步：新增或修改资源条目
在 data/seeds/ 目录下按照模板格式新增或修改对应的资源条目文件。每个条目需包含必填字段：资源标题、原始 URL、资源类型、所属分类、标签列表以及摘要描述。新增条目后需运行 npm run validate 校验格式。

第四步：提交变更并编写清晰的提交信息
提交代码前请确保所有测试通过（npm test）。提交信息应简明扼要地说明本次变更的目的和内容，推荐使用 Conventional Commits 规范，例如 docs: add new resource entry for Kafka performance report。

第五步：发起 Pull Request
将特性分支推送到个人 fork 仓库，然后向主仓库的 main 分支发起 Pull Request。PR 描述中请说明本次变更的背景、具体改动内容以及是否涉及破坏性变更。项目维护者会在 3 个工作日内进行审查。

## 常见问题

**问：资源文件中的 URL 地址如果失效了怎么办？**
项目内置了定时可用性探测任务，默认每 24 小时对所有已收录的外部链接发起 HEAD 请求。如果某个链接连续三次探测均返回非 200 状态码，系统会在该资源条目的元数据中标记为 "unreachable" 状态，并在管理后台高亮显示。用户也可以通过 CLI 命令 npm run check -- --url <资源ID> 手动触发单条探测。对于确认失效的链接，维护者会尝试寻找替代链接或移除该条目。

**问：我能否在本地部署时导入自定义的外部链接列表？**
可以。项目支持通过 CSV 或 JSON 格式批量导入外部链接。你需要在 data/import/ 目录下放置符合 schema 定义的文件，然后执行 npm run import -- --format csv --path ./data/import/my-list.csv 命令。导入前请确保文件中的分类名称与现有分类体系匹配，否则系统会提示创建新分类或跳过条目。详细的导入格式说明请参考 docs/admin/batch-import.md。

**问：项目中的分类体系是如何设计的，我可以修改吗？**
当前分类体系基于技术领域、资源类型和适用层级三个维度设计，具体定义在 config/taxonomy.yaml 文件中。用户可以在本地 fork 或私有部署中自由修改该文件以适配自己的知识管理需求。但如果希望将修改合并到上游主仓库，需要先在 issue 中提出分类变更提案，经过社区讨论并达成初步共识后再提交 PR，以保证分类体系的一致性与可维护性。
