# CSUtkg Reference Index

CSUtkg Reference Index 是一个面向技术研究者和开发人员的结构化外链与资源汇总工具。该项目通过系统化的 Markdown 文档索引，将分散在网络各处的技术文章、教程、规范文档和代码示例进行集中管理和分类导航，解决开发人员在查阅技术资料时面临的链接散乱、检索困难、上下文丢失等问题。

项目定位为轻量级的技术资源导航站，适用于个人开发者构建知识库、技术团队沉淀文档、以及开源项目维护者整理外部参考资料。CSUtkg 不存储实际内容，仅提供链接的规范化整理与分类索引，所有原始资源均托管于第三方平台。

## 功能概览

**链接分类索引**：按照技术领域、文档类型、内容主题对原始链接进行多维度标签分类，支持快速筛选定位。

**Markdown 原生渲染**：所有索引数据以纯 Markdown 格式存储，兼容 GitHub、GitLab、Gitee 等主流代码托管平台的自动渲染。

**哈希版本追踪**：每个资源条目记录文档哈希值，便于追踪外部内容的版本变化和更新状态。

**本地离线查阅**：完整仓库克隆后可在本地环境中直接浏览全部索引内容，无需依赖网络连接。

**批量导入导出**：支持将资源列表以结构化格式批量导入或导出，方便与其他知识管理工具进行数据交换。

**自定义分类体系**：用户可根据自身需求修改分类标签和目录结构，实现个性化资源组织。

**全文检索支持**：配合 grep 或 IDE 内置搜索功能，可对全部索引内容进行快速关键词检索。

**多层级目录导航**：建立三级目录结构（领域/子领域/文档），模拟文件系统的浏览习惯。

## 应用场景

技术文档归档与整理：技术团队在日常开发过程中会产生大量外部参考链接，包括第三方库文档、API 规范、算法讲解视频等。CSUtkg 提供统一的索引模板和分类规范，帮助团队将分散的链接按照项目模块或技术栈进行归集，避免重要资料丢失。

开源项目外部参考管理：开源项目维护者需要在 README 或贡献指南中引用大量外部资源，如编码规范、设计模式讲解、依赖库文档等。使用 CSUtkg 可以建立独立的外部参考索引文件，保持主文档的简洁性，同时为贡献者提供完整的背景资料入口。

个人技术知识体系建设：开发者可以借助 CSUtkg 构建个人技术知识图谱，将日常阅读的技术博客、会议演讲录像、论文预印本等资源按照学习路径进行组织，形成可复用的个人知识资产。

离线文档中心部署：在内部网络或隔离环境中，CSUtkg 可作为离线文档中心的导航入口，配合本地镜像或内网代理，为无法访问外网的开发环境提供资源定位服务。

技术培训课程资料包：培训讲师可以将课程中涉及的全部外部阅读材料、练习题来源、扩展阅读清单通过 CSUtkg 进行汇总，学员一次性克隆仓库即可获得完整的课程资源索引。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 Git Bash 或 WSL2 终端执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg

# 安装依赖工具（基于 Node.js 环境）
npm install -g markdown-link-check @actions/glob

# 运行本地索引构建脚本
./scripts/build_index.sh --source ./main --output ./dist
```

首次运行构建脚本时，程序会遍历 main 目录下的全部 Markdown 文件，提取文档元数据并生成总索引表。该过程约耗时 10 至 30 秒，具体取决于网络状况和文件数量。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | 16.x 或更高 | 用于运行索引构建脚本和链接检查工具 |
| npm | 8.x 或更高 | 包管理器，用于安装自动化工具依赖 |
| Git | 2.30 或更高 | 版本控制，用于克隆仓库和管理更新 |
| Bash | 4.0 或更高 | 运行构建脚本（Windows 用户需 Git Bash） |
| grep | 3.0 或更高 | 全文检索功能的底层依赖 |
| curl | 7.68 或更高 | 用于链接有效性检测（可选功能） |
| markdown-link-check | 3.11 或更高 | 外部链接可用性验证工具 |
| yaml-lint | 1.4 或更高 | 分类标签配置文件语法检查 |

## 文档导航

| 层面 | 目录 | 回答的问题 |
|---|---|---|
| 顶层索引 | main/INDEX.md | 全部资源的分类总览，按技术领域分章节展示链接数量与最近更新时间 |
| 领域分类 | main/domains/ | 每个技术领域（如前端、后端、DevOps）下设独立目录，该目录下存放该领域全部资源的索引文件 |
| 哈希索引 | main/hashes/ | 以文档哈希值为文件名存放原始 Markdown 文件，用于版本追踪和去重校验 |
| 标签映射 | main/tags/ | 按标签（如 tutorial、spec、best-practice）组织的交叉索引，便于跨领域查找 |

## 贡献指南

我们欢迎社区贡献者参与 CSUtkg 项目的建设。贡献形式包括但不限于新增资源链接、更新失效 URL、优化分类标签、改进构建脚本等。请遵循以下步骤：

第一步：Fork 本仓库至个人账户，在本地克隆 Fork 后的仓库，并创建新的功能分支（命名格式为 feature/简述修改内容，例如 feature/add-python-resources）。

第二步：在 main/hashes/ 目录下新增或修改 Markdown 文件。每个文件必须包含标准化的 frontmatter 头部，包括 title、source_url、category、tags、hash 五个字段。新增资源时需运行 ./scripts/update_hashes.sh 计算文档哈希值。

第三步：提交变更并推送到个人 Fork 仓库，随后向本仓库的 main 分支发起 Pull Request。请在 PR 描述中清晰说明修改内容、涉及资源数量以及是否经过本地构建测试。

第四步：等待维护者进行 Code Review。维护者将检查链接有效性、分类合理性和文档格式规范性。若通过审核，PR 将被合并至 main 分支。

第五步：合并后，GitHub Actions 将自动触发全量链接检查，若发现失效链接，维护者会创建 Issue 并标记 help-wanted，社区贡献者可认领修复任务。

## 常见问题

问：该项目是否存储原始文档的完整内容？

答：CSUtkg 仅存储外部资源的元数据（标题、来源 URL、标签、摘要描述）和文档哈希值，不存储原始文档的完整正文内容。所有原始资源均托管于第三方平台，用户需通过资源列表中的链接访问原始文档。

问：如何报告失效链接或请求新增资源？

答：请在 GitHub Issues 中选择对应的模板（Broken Link Report 或 Resource Request），按照模板要求填写失效链接的原始 URL 和所在文件路径，或提供新增资源的名称、URL 和推荐分类。维护者会定期处理 Issue 并更新索引。

问：索引构建脚本在 Windows 环境下无法运行怎么办？

答：Windows 用户推荐使用 Git Bash（随 Git for Windows 安装）执行构建脚本。若遇到路径分隔符问题，可将脚本中的路径变量统一修改为双引号包裹形式。项目计划在 v2.0 版本中提供跨平台的 Node.js 实现版本以替代 Bash 脚本。
