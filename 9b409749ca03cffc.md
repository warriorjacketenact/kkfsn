# CSUTKG External Resource Index

CSUTKG External Resource Index 是一个面向技术文档工程师、开源项目维护者以及知识库管理员的静态资源外链汇总系统。该项目旨在解决分散于多个仓库、多个分支中的技术文档、配置文件、变更日志等 Markdown 文件难以统一检索与引用的问题。通过建立基于哈希索引的外链映射表，CSUTKG 允许用户将散落在任意 Git 仓库中的单篇 Markdown 文档进行集中式登记、分类标注和版本关联，从而形成可追溯、可校验的外部知识库引用网络。

本项目并非传统的爬虫或全文搜索引擎，而是一个轻量级的前端资源导航与元数据管理工具。它接收外部仓库中的原始 Markdown 文件路径，提取其哈希标识与提交上下文，生成稳定的永久引用链接，并按照技术领域、文档类型、更新时间等维度进行分组展示。项目主要服务于需要维护大量跨仓库技术文档的外部引用关系的团队，例如 SDK 发布说明聚合、多模块配置项总览、微服务架构下的接口定义文档索引等场景。通过本项目提供的静态站点生成能力，用户可以将零散的技术文档汇聚为一个结构清晰、可离线访问的知识图谱入口。

## 功能概览

**基于哈希的永久外链映射**：每个收录的外部 Markdown 文档均以其文件内容哈希或提交 SHA 作为唯一标识，生成不随原仓库分支变动而失效的稳定访问路径。

**多维度分类标签系统**：支持为每条外链记录添加技术栈、文档类型、所属模块、维护团队等自定义标签，便于按业务场景快速筛选。

**批量导入与目录树同步**：支持通过命令行工具批量导入来自同一仓库或不同仓库的文档链接，并自动与本地项目目录树进行同步校验，确保无死链。

**静态站点生成器集成**：内置基于 Go 模板的静态页面生成引擎，可将所有外链记录输出为具有搜索框、标签云和时间轴导航的纯 HTML 静态站点。

**变更历史追踪**：记录每条外链的添加时间、最后校验时间和内容哈希变化，当原文档发生更新时提供视觉差异提示。

**访问统计与健康检查**：周期性对已收录的外链进行 HTTP HEAD 请求探测，标记失效链接并生成健康报告。

**权限分级视图**：支持配置公开只读视图与内部可编辑视图，满足开源项目公开文档索引和团队内部知识库管理的双重需求。

## 应用场景

**多模块微服务接口文档聚合**：当技术团队维护数十个微服务仓库，每个仓库均包含独立的 API 文档 Markdown 文件时，使用 CSUTKG 可将所有接口说明汇总至统一入口，开发者只需访问该入口即可查阅全部服务的端点定义、错误码表和版本变更记录。

**开源项目外部贡献指南索引**：大型开源社区往往在多个仓库中存放贡献者指南、行为准则和本地开发环境配置说明。CSUTKG 可将这些分布式的指南文档集中收录，并按照仓库名称或功能领域分类展示，降低新贡献者的学习成本。

**配置项与环境变量全局参考**：对于拥有复杂配置体系的系统，不同微服务或组件的配置项说明可能散落在各自仓库的 README 或 docs 目录中。通过 CSUTKG 统一收录，可以形成一份跨服务的配置项总览表，便于运维人员在变更环境变量时快速查阅所有相关服务的配置影响范围。

**技术规范与设计文档版本追溯**：在敏捷开发过程中，技术设计文档、数据库 Schema 变更说明和架构决策记录通常分布在多个代码仓库的特定分支中。CSUTKG 记录的哈希引用和提交上下文可帮助团队精确追溯某一设计决策对应的原始文档版本，避免因分支合并导致的文档错乱。

## 快速开始

以下命令演示了如何从 GitHub 克隆 CSUTKG 项目、安装依赖并启动本地开发服务器。

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg
npm install
npm run build
npm start
```

执行上述命令后，本地服务将默认监听 8080 端口。访问 http://localhost:8080 即可看到已收录的外链资源列表。若需重新生成静态站点，请执行 `npm run generate`，输出目录为 `./dist`。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Node.js | v18.17.0 或更高 | 项目运行时环境，用于执行构建脚本和本地服务 |
| npm | v9.6.0 或更高 | 包管理器，用于安装项目依赖 |
| Git | v2.30.0 或更高 | 用于克隆仓库和获取外部文档的提交元数据 |

| SQLite | v3.35.0 或更高 | 本地元数据存储引擎，用于记录外链映射和变更历史 |
| curl | v7.68.0 或更高 | 用于健康检查模块中的 HTTP 探测（可选，若使用 Node.js 原生 http 模块则可忽略） |

## 贡献指南

1. 在 GitHub 上 Fork 本仓库至个人账户，随后将 Fork 后的仓库克隆至本地开发环境。请确保本地 Git 全局配置了正确的 user.name 和 user.email，以便提交记录可被正确追溯。

2. 新建一个以 `feat/` 或 `fix/` 为前缀的特性分支，例如 `feat/add-tag-filter`。在该分支上进行代码修改或文档更新。若涉及新增依赖，请同步更新 `go.mod` 和 `go.sum` 文件，并在 `docs/developer-guide/architecture.md` 中补充相关说明。

3. 所有代码变更必须通过单元测试和集成测试。在提交 Pull Request 之前，请于项目根目录执行 `go test ./... -cover` 确保测试通过且覆盖率不低于 85%。若新增功能模块，需同步编写对应的测试用例。

4. 提交信息请遵循 Conventional Commits 规范，使用 `feat:`、`fix:`、`docs:`、`chore:` 等类型前缀，并在提交正文中详细描述变更动机、实现方式及影响范围。Pull Request 标题需以对应类型前缀开头，并在描述中引用相关 Issue 编号（若有）。

5. 提交 Pull Request 后，项目维护者将进行 Code Review。请根据反馈及时调整代码或补充测试。合并前需确保所有 CI 检查（包括代码风格检查、静态分析和安全扫描）均通过。

## 常见问题

**问：如果原仓库中的 Markdown 文件被重命名或移动，已收录的外链是否会失效？**

答：CSUTKG 的外链映射基于文件内容的哈希值（即 URL 路径中的十六进制字符串），而非文件路径或文件名。只要原文件的内容不发生改变，即使文件被移动到其他目录或仓库，用户只需在系统后台更新该记录对应的原始路径字段，已生成的永久链接仍可正常访问。若文件内容发生实质变更，系统会将其识别为新版本，并在变更历史中保留旧版本的存档链接。

**问：静态站点生成后的 HTML 文件能否部署到 GitHub Pages 或 Nginx 等静态托管服务？**

答：可以。执行 `npm run generate` 或 `go run cmd/generator/main.go` 后，所有 HTML、CSS、JavaScript 资源均输出至 `./dist` 目录，该目录完全包含独立的静态文件，无需任何后端服务支持。您可以将 `./dist` 目录下的全部内容上传至任意静态托管服务。但需注意，搜索功能和标签筛选依赖前端 JavaScript 在浏览器端对已生成的 JSON 索引文件进行内存过滤，对于超过 5000 条外链的大型数据集，建议启用服务端分页或使用预过滤的静态页面生成策略。

**问：健康检查模块发现失效链接后会如何处理？**

答：健康检查模块会按照 `configs/default.yaml` 中配置的间隔（默认每 24 小时）对所有已收录外链进行并发探测。若某条链接连续三次返回非 200 状态码或超时，系统会将该链接标记为 `unhealthy`，并在下一次生成的健康报告中以醒目标识展示。同时，系统会向配置的告警 Webhook（支持 Slack、钉钉和邮件）发送通知。用户可在管理界面手动重新验证被标记的链接，或更新其原始路径以恢复可用状态。被标记为不健康的链接不会从索引中自动删除，但会在前端界面显示警告图标。
