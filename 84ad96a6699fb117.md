# CSU-TKG Technical Knowledge Graph

CSU-TKG is a curated technical knowledge graph and resource aggregation system that indexes, categorizes, and cross-references distributed technical documentation, code snippets, and architectural decision records. The project targets software architects, DevOps engineers, and technical leads who need to maintain a navigable map of external references across large-scale development initiatives.

The system operates as a lightweight static site generator that consumes markdown-based reference files from a structured repository and produces a unified knowledge graph interface. It solves the problem of fragmented external documentation by providing a single entry point to 180+ indexed reference nodes, each tagged with metadata for fast retrieval and dependency tracking.

## 功能概览

**Automated Reference Harvesting** - Scans designated repository paths for markdown files containing embedded metadata headers and extracts key-value pairs for indexing.

**Dependency Graph Generation** - Parses cross-file references and produces a directed acyclic graph (DAG) that visualizes relationships between technical decision records and implementation notes.

**Tag-Based Filtering System** - Assigns multiple taxonomy tags to each resource (e.g., architecture, security, deployment, performance) and enables faceted search across the entire corpus.

**Version History Tracking** - Maintains a changelog for each indexed resource by comparing file hashes across Git commits, highlighting modifications and deprecation warnings.

**Static Site Compilation** - Generates a fully offline-compatible HTML dashboard with search, filter, and detail views, requiring no runtime database or external API calls.

**RESTful JSON API** - Exposes all indexed metadata and graph relationships via a read-only HTTP endpoint for integration with external monitoring dashboards and CI/CD pipelines.

**Batch Import Pipeline** - Supports bulk ingestion of resource lists through a command-line interface that validates URL formats, checks accessibility, and assigns initial category tags.

## 应用场景

**Technical Onboarding for New Team Members** - New engineers joining a project can browse the knowledge graph to understand which external references inform critical architectural decisions, reducing ramp-up time by providing immediate context for design choices.

**Pre-Implementation Architecture Review** - During planning phases, architects use the system to check whether proposed solutions duplicate existing patterns or conflict with documented constraints, surfacing relevant references before code is written.

**Incident Post-Mortem Correlation** - When production incidents occur, responders query the graph to locate all decision records and external links that relate to the affected subsystem, ensuring that remediation steps account for previously documented assumptions.

**Dependency Upgrade Impact Analysis** - Before bumping library versions, teams scan the knowledge graph to identify which reference documents rely on the current version, allowing them to test and update affected documentation in coordination with code changes.

## 快速开始

Clone the repository, install dependencies, and run the ingestion pipeline using the commands below.

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg
pip install -r requirements.txt
python ingest.py --batch ./resources/batch_56_76.txt
python build.py --output ./dist
```

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Python | 3.9 及以上 | 核心运行环境，负责解析、索引和构建流程 |
| Git | 2.30 及以上 | 用于版本追踪、文件哈希比对和变更日志生成 |
| Pandoc | 2.10 及以上 | 将扩展 markdown 语法转换为 HTML 内容块 |
| PyYAML | 6.0 及以上 | 解析资源文件中的 YAML frontmatter 元数据 |
| NetworkX | 2.8 及以上 | 构建和操作依赖图数据结构，支持拓扑排序 |
| Jinja2 | 3.1 及以上 | 模板引擎，用于渲染静态 HTML 页面 |
| Pytest | 7.0 及以上 | 单元测试和集成测试框架（开发依赖） |
| pre-commit | 3.0 及以上 | Git 钩子管理，用于代码质量检查（开发依赖） |

## 项目结构

```
csutkg/
├── ingest/                          # 资源导入与预处理模块
│   ├── harvesters/                  # 各类资源采集器实现
│   │   ├── github_harvester.py      # 从 GitHub 仓库抓取 markdown 文件元数据
│   │   └── local_fs_harvester.py    # 从本地文件系统递归扫描资源
│   ├── parsers/                     # 元数据解析器
│   │   ├── frontmatter_parser.py    # 解析 YAML 前置数据块
│   │   └── link_extractor.py        # 提取并验证资源间引用链接
│   └── validators/                  # 资源校验工具
│       └── url_health_check.py      # 验证 URL 可访问性及响应状态
├── core/                            # 知识图谱核心逻辑
│   ├── graph/                       # 图数据结构与算法
│   │   ├── directed_graph.py        # 有向图实现，支持循环检测
│   │   └── topological_sort.py      # 拓扑排序算法输出依赖顺序
│   ├── indexes/                     # 倒排索引与标签系统
│   │   ├── inverted_index.py        # 标签-资源倒排索引构建
│   │   └── tag_normalizer.py        # 标签别名映射与标准化
│   └── cache/                       # 缓存与版本管理
```

## 贡献指南

3.  运行本地验证套件 `pytest tests/unit/` 确保现有功能未被破坏，新增资源导入后执行 `python ingest.py --validate` 检查元数据格式是否符合规范。

4.  提交代码时需通过 pre-commit 钩子检查，包括 YAML 格式校验、Markdown 链接有效性检查以及 Python 代码风格检查（flake8 和 black）。

5.  发起 Pull Request 到主分支，描述中需注明本次新增资源的批次编号和简要变更摘要，项目维护者将在两个工作日内进行审查。

## 常见问题

**Q: 资源链接指向的 markdown 文件内部必须包含哪些元数据字段？**

A: 每个资源文件必须在文件顶部包含 YAML frontmatter 块，至少定义 `title`、`category`、`tags` 和 `created_at` 四个字段。`category` 字段应从预定义分类列表中选取，`tags` 为列表类型。缺少任一字段会导致导入流程跳过该文件并输出警告日志。

**Q: 如何批量更新已导入资源的标签分类？**

A: 系统不支持直接修改已导入资源的标签。正确做法是修改源 markdown 文件的 frontmatter 中的 `tags` 字段，然后运行 `python ingest.py --update`，该命令会重新计算所有文件的哈希值并仅更新变更部分。更新完成后需要重新执行 `build.py` 以刷新静态站点内容。

**Q: 静态站点生成后，搜索功能无法返回任何结果，可能的原因是什么？**

A: 最常见的原因为构建过程中未正确生成倒排索引文件。请检查构建日志中是否包含 `Inverted index built successfully` 字样。若日志显示索引为空，则可能是资源文件中 `tags` 字段解析失败，应验证所有资源的 frontmatter 格式是否正确。另可尝试删除 `build/static/index.json` 后重新执行完整构建流程。
