# CSU Tech Knowledge Gateway (csutkg)

CSU Tech Knowledge Gateway (csutkg) 是一个面向技术研究者、开发者和开源爱好者的外链资源汇总与知识索引系统。项目定位为技术文档的元目录和结构化导航工具，主要解决个人知识管理中存在的信息碎片化、书签散落、回顾困难等问题，通过将分散于各类仓库、博客、教程站点中的高价值技术文档按统一格式收录并生成可追溯的永久化引用路径，帮助用户构建属于自己的技术知识参照体系。

项目本身不托管文档原文，而是以严格格式化的 Markdown 引用链路为核心资产，每一篇收录文档均以独立的 .md 文件存在于仓库中，文件命名采用内容寻址的哈希方式以保证唯一性和可追溯性。适合作为技术团队内部知识库的底层骨架，也适合个人开发者作为学习路径的公开记录。

## 功能概览

- **结构化外链收录** 每一篇文档以独立 Markdown 文件存放，内容包含原文标题、原始出处 URL、收录日期、关键词标签和摘要说明，便于检索和回溯。

- **哈希寻址命名** 所有收录文件使用 16 位十六进制哈希作为文件名，避免重名冲突，同时支持通过哈希值快速定位特定条目。

- **多维度分类索引** 通过目录结构、标签系统和文档导航表格实现按技术领域、文档类型、适用场景等多维度筛选。

- **自动化文档模板** 提供标准化的 Markdown 模板，确保所有收录条目保持一致的元数据字段和排版风格。

- **版本追踪友好** 每条收录记录可独立追踪更新历史，通过 Git 提交记录了解文档的添加或修订时间。

- **批量导入支持** 提供脚本工具将书签文件、浏览器收藏夹或 CSV 清单批量转换为符合规范的收录文档。

- **离线阅读准备** 所有收录内容均为纯 Markdown 格式，支持在不联网环境下通过本地编辑器或静态站点生成器查看。

- **社区贡献流程** 开放 Pull Request 和 Issue 提交通道，允许社区成员补充、修正或推荐新的技术资源链接。

## 应用场景

- **个人技术学习路径归档** 开发者可将日常阅读的教程、API 参考、架构设计文档按主题分类收录，形成可随时查阅的个人学习手册，避免重复搜索。

- **团队内部知识库基础** 技术团队可基于本项目模板建立私有分支，将内部系统文档、第三方服务对接指南、故障处理记录统一纳入索引，降低信息孤岛问题。

- **开源项目外部依赖记录** 开源项目维护者可利用本仓库记录项目中引用的外部资源、依赖库官方文档和相关技术提案，便于新贡献者快速理解项目背景。

- **技术资讯与前沿论文跟踪** 研究人员可定期将 arXiv 预印本、技术博客、会议录像链接收录归档，配合标签系统实现按研究方向和时间线的筛选。

- **离线文档站点的数据源** 配合静态站点生成器（如 Hugo、VuePress）可将收录的 Markdown 文件生成为完整的离线文档网站，适合内网部署或移动端阅读。

## 快速开始

以下命令序列演示如何从 GitHub 克隆本仓库、安装基础依赖并运行本地预览服务。

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg

# 安装 Node.js 依赖（用于本地预览和文档校验）
npm install

# 启动本地开发服务器，默认监听 http://localhost:3000
npm run dev
```

执行完毕后，浏览器访问本地地址即可查看收录文档的索引首页和分类导航。若需仅使用命令行模式进行文档校验或批量导入，可执行 `npm run build` 生成静态文件，或 `npm run lint` 检查所有收录文档的格式规范。

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js 18.x 或更高版本 | 是 | 运行本地开发服务器、构建脚本及文档校验工具 |
| npm 9.x 或更高版本 | 是 | 管理项目依赖包及执行脚本命令 |
| Git 2.30 或更高版本 | 是 | 克隆仓库、提交更新及版本回溯 |
| markdownlint-cli | 否 | 用于检查收录文档的 Markdown 语法规范性，推荐安装 |
| rimraf | 否 | 清理构建缓存目录，仅在执行完整重建时需要使用 |
| http-server | 否 | 轻量级静态文件服务，可在无 Node.js 环境下替代 dev 服务器 |
| Python 3.9+（可选） | 否 | 仅在运行旧版导入脚本时需要，新版已迁移至 Node.js |

## 贡献指南

所有形式的贡献均视为对本项目使用协议（MIT License）的同意。贡献前请确保已阅读并理解上述文档导航部分所列的规范说明。

1. 复刻本仓库至个人账户，并克隆至本地开发环境。创建新的功能分支，分支命名建议使用 `feat/` 或 `fix/` 前缀加简要描述。

2. 新增收录文档时，请使用 `scripts/template-generator.js` 生成标准模板，填写完整的元数据字段（标题、原始URL、标签、摘要），确保 Markdown 格式符合 `.markdownlint.json` 中定义的规则。

3. 批量添加或修改文档后，在本地运行 `npm run lint` 和 `npm run build` 以验证格式和构建流程无误。所有新提交的文件名必须为 16 位小写十六进制字符串。

4. 提交变更时，撰写清晰的 commit message，格式为 `<type>(<scope>): <subject>`，例如 `feat(main): add 180 links for batch 16`。推送分支后，在 GitHub 上发起 Pull Request 到主分支。

5. Pull Request 描述中请说明变更内容、影响范围以及是否涉及破坏性改动。至少一名项目维护者审核通过后即可合并。

## 常见问题

**问：收录文档的哈希文件名是如何生成的？是否与内容相关？**

答：哈希文件名是在导入时由脚本根据当前时间戳和随机数组合生成的 16 位十六进制字符串，用于确保文件名唯一且不重复。该哈希与文档的实际内容或原始 URL 没有加密意义上的关联，不用于内容寻址或完整性校验。如果需要验证文档内容的完整性，建议配合 Git 的 commit SHA 进行版本追踪。

**问：如果原始资源链接失效或者内容被删除，本仓库如何处理？**

答：本仓库仅收录原始资源的 URL 元数据，不托管原文内容。因此原始链接失效属于上游问题，项目本身无法修复。维护者会定期通过 CI 脚本检查收录链接的 HTTP 状态码，对于返回 404 或 410 的链接，会在对应的收录文档中添加 `[dead-link]` 标记并记录检查时间，但不会主动删除条目，以保留引用上下文。用户可使用 `npm run check-links` 手动触发链接有效性检查。

**问：如何批量添加我自己收藏的技术文档链接？**

答：将链接列表整理为 CSV 文件，每行包含 `title, url, tags, summary` 四个字段，然后运行 `node scripts/import-from-csv.js --file your-list.csv`。脚本会自动为每一行生成符合规范的 Markdown 文档，并输出新增文件的哈希列表。导入前建议先用 `--dry-run` 参数预览生成结果。
