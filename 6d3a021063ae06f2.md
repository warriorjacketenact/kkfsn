# TekStack Knowledge Gateway

TekStack Knowledge Gateway 是一个面向开发者与技术研究人员的结构化技术资源外链聚合与导航系统。该项目并非传统的代码库或框架，而是一个精心编排的知识索引工程，旨在解决技术信息碎片化、优质资源分散、检索成本高的问题。项目通过统一的 Markdown 元数据描述与分类体系，将散落在网络各处的深度技术文章、工具文档、架构方案与工程实践整理为可追溯、可交叉引用的知识图谱。

项目定位为轻量级的技术资源中台，不依赖数据库与后端服务，完全基于静态 Markdown 文件与 GitHub 仓库的版本管理能力运行。目标用户包括软件工程师、架构师、技术决策者、开源贡献者以及计算机科学领域的研究人员。项目核心价值在于提供一套可私有化部署、可社区共建的外链数据规范，使得每个技术团队都能快速建立属于自己的知识导航站，同时共享主仓库的持续更新与审核成果。

## 功能概览

**结构化外链索引**：所有资源链接按照技术领域、难度等级、内容类型、更新时间四个维度进行标准化标注，每条记录均附带内容摘要与关键词标签，便于后续检索与过滤。

**Markdown 元数据驱动**：每个外链条目对应一个独立的 Markdown 文件，文件头部采用 YAML Front Matter 格式记录资源属性，正文部分可存储详细的阅读笔记与点评，实现链接与知识沉淀的有机结合。

**自动化目录生成**：基于文件命名规范与元数据标签，系统可自动生成多层次分类目录，包括按编程语言、框架生态、应用场景、作者机构等不同视角的导航页，无需人工维护索引表。

**全文检索与标签过滤**：通过集成 GitHub Pages 支持的静态站点生成器或本地脚本工具，可实现对外链标题、摘要、标签、正文笔记的即时搜索，支持多标签组合筛选，显著降低查找成本。

**版本追踪与失效检测**：利用 Git 的提交历史与定时检查脚本，项目能够追踪每条外链的添加、更新与删除记录，并定期对已收录链接进行可达性检查，标记失效或内容变更的资源，保证索引的长期有效性。

**社区共建工作流**：提供完整的外部贡献者指南与 Pull Request 审核模板，允许社区成员提交新的资源链接、更新已有条目信息或提出分类调整建议，所有变更均经过维护者审核与元数据校验。

**多格式数据导出**：支持将整个外链索引导出为 JSON、CSV、RSS Feed 等通用数据格式，方便与其他工具链集成，例如导入到个人知识管理软件、团队 Wiki 或自动化报告系统中。

## 应用场景

团队技术文档库的补充导航层。当企业内部已有 Confluence 或 Notion 等文档系统时，TekStack Knowledge Gateway 可作为外部优质资源的前置筛选层，帮助团队成员快速定位与当前项目相关的第三方库文档、解决方案博客与性能调优案例，减少重复造轮子与无效搜索时间。

技术博客与个人知识管理站点的资源附录。独立博主或知识管理爱好者可使用该项目作为其文章的外部引用库，将散落在各篇博文中的超链接统一收录并进行分类管理，既提升文章的整洁度，也方便读者按专题批量获取延伸阅读材料。

开源项目官方文档的生态资源页。开源软件维护者可将 TekStack Knowledge Gateway 作为官方文档的一个独立章节或子站点，集中展示社区推荐的插件、工具链、部署方案与最佳实践文章，降低新用户的上手门槛并促进生态的活跃发展。

技术培训与课程设计的教学素材库。教育工作者或企业培训部门可基于该项目构建课程相关的延伸阅读清单，按照教学周次或知识点模块组织外链，学生可通过统一的索引页面获取所有参考资料，同时教师可方便地更新和版本化管理教学资源。

技术社区的内容聚合与推荐系统。技术社区运营团队可将该项目作为社区内容推荐的底层数据源，通过定期合并社区成员提交的资源链接，生成每周优质内容摘要、月度技术盘点等衍生内容，丰富社区的互动形式与内容生态。

## 快速开始

以下指令适用于 Linux、macOS 及 Windows WSL 环境，需提前安装 Git 与 Node.js 运行时。

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg
npm install
npm run build
```

执行完成后，可在 `dist` 目录下获得静态生成的完整导航站点文件，使用任意 HTTP 服务器（如 nginx、serve、Python http.server）即可启动本地预览。若仅需使用外链数据而不需要站点界面，可运行 `npm run export:json` 获得机器可读的索引数据。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Git | 2.25.0 或更高 | 用于克隆仓库、提交更新、查看历史记录与分支管理 |
| Node.js | 18.0.0 LTS 或更高 | 项目构建脚本、元数据校验器与导出工具均基于 Node.js 运行时 |
| npm | 8.0.0 或更高 | 用于安装项目依赖包，包括 Markdown 解析器、模板引擎与测试框架 |
| Markdown 编辑器 | 任意 | 推荐使用支持 YAML Front Matter 语法高亮的编辑器，如 VS Code 配合 markdownlint 插件 |
| 网络连接 | 稳定 | 构建过程中无需在线请求，但提交新外链时需要访问目标页面以核实内容与元数据准确性 |


## 贡献指南

提交新资源链接之前，请先阅读 `docs/contributor/` 下的完整指引，并确保已通过本地构建测试。所有贡献均需遵循元数据 Schema 定义，否则自动化校验将拒绝合并。

第一步：从 main 分支创建新的功能分支，分支命名格式为 `add/resource-{short-description}` 或 `update/{entry-hash}`。确保本地环境已安装所有依赖，并且 `npm run test` 能够通过。

第二步：在 `entries/_raw/` 目录下创建新的 Markdown 文件，文件名沿用项目规定的哈希命名方式。文件头部必须包含完整的 YAML Front Matter，至少包括 title、url、tags、difficulty、source 五个字段。正文部分可自由撰写推荐理由、内容摘要或阅读笔记。

第三步：运行 `npm run validate` 对新添加的文件进行元数据校验，根据错误提示调整字段内容。校验通过后，运行 `npm run build` 在本地生成完整的导航站点，手动检查新条目是否出现在正确的分类下，并确认链接指向的页面内容与描述一致。

第四步：提交变更并推送到远程分支，随后在 GitHub 上创建 Pull Request。PR 描述中需说明该资源的来源、推荐理由以及适用于哪些技术场景。维护者会在三个工作日内进行审核，必要时会通过评论请求补充信息或修改。

第五步：PR 被合并后，CI 流水线将自动重新构建生产站点并更新在线导航页。贡献者名称会被记录在对应条目的 `contributor` 元数据字段中，并在站点首页的贡献者列表中展示。

## 常见问题

问：外链目标页面发生内容变更或直接失效时，项目如何处理？

答：项目维护者每月通过 GitHub Actions 定时任务运行链接检测脚本，对所有收录链接进行 HTTP 状态码与页面标题比对。对于状态码非 200 或标题发生显著变化的条目，系统会自动创建 Issue 并标记为 `link-check` 标签。维护者根据 Issue 信息核实并决定是更新描述、替换链接还是移除条目。社区贡献者也可以通过手动运行 `npm run check:links` 触发本地检测，并将结果提交为 Issue 或 PR。

问：如果我想在项目基础上增加自己的分类体系，是否需要修改核心代码？

答：不需要修改核心代码。项目的分类体系完全由 `entries/` 目录下的软链接视图和元数据中的 `tags` 字段共同驱动。您可以在 `entries/by-topic/` 下创建新的符号链接目录指向 `_raw/` 中的原始文件，或通过修改 `src/build/tagMapping.js` 中的分类映射表来调整渲染时的分组逻辑。对于个人使用场景，更推荐直接复制整个项目并修改 `src/build/config.js` 中的分类配置，而不向主仓库提交此类个性化变更。

问：项目的静态站点生成过程是否依赖外部网络或 API？

答：不依赖。整个构建过程完全本地化，所有数据来源于 `entries/` 目录下的静态 Markdown 文件。模板引擎采用预编译方式，无需在构建时请求任何在线资源。导出 JSON 与 RSS Feed 的功能同样基于本地数据生成。唯一需要网络连接的场景是运行链接检测脚本时，该脚本会向外链目标发送 HTTP 请求以验证可用性。
