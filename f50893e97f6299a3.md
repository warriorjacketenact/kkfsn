# CSUtkg Resource Indexer

CSUtkg Resource Indexer 是一个面向技术社区的开源资源聚合与导航系统，专注于收集、分类和呈现分散在代码仓库中的技术文档、配置示例、问题排查记录与架构设计备忘。该项目并非传统的包管理工具或框架脚手架，而是一个轻量级的知识索引层，帮助开发者在大规模代码库中快速定位具有参考价值的片段化内容。

项目主要服务于以下三类用户：日常维护多个代码仓库的架构师与 Tech Lead，需要从既有实现中提炼通用模式的中高级开发工程师，以及希望系统化学习特定技术栈（如 Go 运行时、分布式共识协议、云原生可观测性）的研究型开发者。CSUtkg Resource Indexer 本身不存储业务数据，而是通过稳定 URL 引用机制，将外部仓库中的 Markdown 文档、配置片段与流程图描述聚合为统一的可检索目录，从而降低团队内部知识传递的隐性成本。

## 功能概览

**多级标签体系** 每条资源可附加多个领域标签，支持跨技术栈的联合筛选，例如同时标记 "etcd" 与 "gRPC 调优" 的资源可被独立归入分布式存储子类。

**结构化元数据抽取** 从资源文件的头部注释或特定标记块中自动提取作者、创建时间、关联 Issue 编号与依赖组件版本，形成标准化的卡片视图。

**全文检索引擎集成** 基于 Bleve 实现轻量级本地索引，支持对资源标题、正文片段与代码注释的模糊检索，检索结果按相关性降序排列。

**批量导入与增量更新** 提供命令行工具，可从给定的 URL 清单批量拉取元数据，并在源仓库更新后通过 Webhook 触发增量刷新，无需全量重建。

**资源依赖拓扑可视化** 针对配置类文件（如 docker-compose.yml、Kustomization 清单），自动解析其引用的外部镜像、卷挂载与 Secret 依赖，生成依赖关系有向图。

**历史快照对比** 记录每次资源更新的哈希值，支持任意两个版本之间的差异对比，并以 unified diff 格式输出变更摘要，便于审计。

**导出为结构化文档** 支持将筛选后的资源列表导出为 Markdown 表格、JSON 或 YAML 格式，方便嵌入团队 Wiki 或 CI/CD 流水线文档。

## 应用场景

**技术选型调研** 当团队计划引入新的中间件（如 Apache Pulsar 或 VictoriaMetrics）时，工程师可通过 CSUtkg Resource Indexer 快速检索已有仓库中与该中间件相关的配置样例、性能测试脚本与故障排查记录，从而评估实际落地的复杂度与潜在风险。

**故障复盘知识库** 运维人员在处理线上事故后，可将临时记录的修复命令、日志截断点与配置回滚步骤提交为 Markdown 资源文件，并通过本索引器添加 "故障类型：网络分区" 和 "严重程度：P1" 等标签，形成可检索的复盘档案。

**代码重构参考** 在进行大规模模块拆分或包重命名时，架构师依赖索引器找出所有引用被修改包的外部资源（包括部署脚本、监控告警规则与集成测试用例），确保重构不会破坏下游依赖。

**新人入职引导** 团队为新成员准备一份包含核心业务流程图、API 设计规范、本地开发环境搭建指南的入门资源清单，通过索引器的固定链接功能生成专属导航页，替代传统的零散文档分发方式。

## 快速开始

以下命令演示如何从主仓库克隆 CSUtkg Resource Indexer，安装依赖并启动本地服务。

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg
go mod download
go run cmd/indexer/main.go --config configs/default.yaml --sync
```

执行上述命令后，索引器会读取 configs/default.yaml 中预设的资源清单路径，将当前批次（第 12/76 批，共 180 个资源链接）的元数据拉取至本地缓存，并启动 HTTP 服务监听 8080 端口。访问 http://localhost:8080/ui 可打开 Web 管理界面。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---|---|---|
| Go 工具链 | 1.21 及以上 | 编译与运行主程序，依赖泛型与 slog 日志模块 |
| Git | 2.30 及以上 | 克隆外部仓库、拉取资源文件变更、计算文件哈希 |
| SQLite | 3.35 及以上 | 存储资源元数据、标签映射与历史快照记录 |
| Bleve 索引库 | v2.3.10 | 全文检索引擎，负责构建倒排索引与查询解析 |
| yq | v4.34 及以上 | 可选依赖，用于解析 YAML 格式的配置文件元数据 |
| Docker | 20.10 及以上 | 仅当启用容器化部署或依赖拓扑可视化功能时需要 |

## 项目结构

项目遵循标准的 Go 单仓库多模块布局，核心逻辑与命令行入口分离，便于后续扩展 gRPC 或消息队列接入层。

```
csutkg/
├── cmd/                                 # 可执行程序入口
│   └── indexer/
│       └── main.go                      # 服务启动、信号处理、配置加载
├── internal/                            # 内部包，不对外暴露
│   ├── crawler/                         # 资源拉取与解析逻辑
│   │   ├── fetcher.go                   # 基于 go-git 克隆与增量拉取
│   │   └── parser.go                    # 提取 Markdown 头部元数据
│   ├── index/                           # 全文索引封装
│   │   ├── bleve_adapter.go             # Bleve 索引初始化、增删改查
│   │   └── query_parser.go              # 查询语法转换与权重调节
│   ├── storage/                         # SQLite 持久化层
│   │   ├── schema.sql                   # 资源表、标签表、快照表 DDL
│   │   └── repository.go                # 基于 database/sql 的 CRUD
│   └── graph/                           # 依赖拓扑可视化
│       ├── resolver.go                  # 解析 docker-compose 与 Kustomize
│       └── dot_exporter.go              # 生成 Graphviz DOT 格式
├── pkg/                                 # 可被外部引用的公共库
│   ├── models/                          # 资源实体、标签、快照结构体
│   ├── config/                          # YAML 配置结构体与默认值
│   └── utils/                           # 哈希计算、时间戳工具函数
├── configs/
│   └── default.yaml                     # 默认端口、索引路径、批次配置
├── docs/                                # 完整文档目录
│   ├── user-guide/
│   ├── operations/
│   ├── developer/
│   └── api/
├── scripts/                             # CI/CD 辅助脚本
│   ├── import-batch.sh                  # 批量导入当前批次 URL 清单
│   └── webhook-listener.py              # 简单 Flask 服务接收 GitHub Webhook
├── testdata/                            # 用于单元测试的样例资源文件
├── go.mod
├── go.sum
└── README.md
```

## 贡献指南

我们欢迎社区以多种形式参与 CSUtkg Resource Indexer 的演进。所有贡献者需遵守行为准则，并在提交变更前阅读开发者文档。

1. 选择未认领的 Issue 或提出新需求。浏览 GitHub Issues 中标记为 "help wanted" 或 "good first issue" 的工作项，在评论区声明认领，避免重复开发。

2. 派生仓库并创建功能分支。从 main 分支切出新分支，命名格式为 feature/功能简述 或 fix/问题编号。确保分支名称简洁且可读。

3. 编写或更新单元测试。所有新增的解析器逻辑、索引查询路径以及存储层方法都必须附带对应的测试文件，覆盖率不低于 80%。运行 make test 验证本地变更。

4. 提交 Pull Request。在 PR 描述中关联对应的 Issue 编号，列出变更摘要、手动测试步骤以及可能影响性能或兼容性的风险点。PR 需要通过所有 CI 检查项，包括 lint、vet 和集成测试。

5. 文档同步更新。若变更涉及配置字段、命令行参数或 HTTP 端点，需同步更新 docs/ 下对应的手册。对于新增的资源字段类型，务必更新元数据抽取规范文档。

## 常见问题

**问：如何更新已导入的资源元数据，例如修正错误的标签或补充缺失的描述字段？**

答：目前索引器不提供在线编辑功能，以保证数据源的单向可信。如需修正，请直接修改原始 Markdown 文件头部的 front-matter 区块，然后通过 Webhook 或手动执行 ./scripts/import-batch.sh --incremental 触发增量同步。索引器会检测文件哈希变化并覆盖旧元数据。

**问：索引器是否支持私有仓库或需要认证的 Git 源？**

答：支持。在 configs/default.yaml 中可配置多个 origin 源，每个源可独立设置 HTTP 基本认证或 SSH 私钥路径。对于 GitHub 私有仓库，建议使用个人访问令牌 (PAT) 并通过环境变量 GITHUB_TOKEN 注入，索引器会在拉取时自动将令牌附加到 HTTPS 请求头。

**问：全文检索为何无法匹配中文或日文关键词？**

答：Bleve 默认的分析器基于 Unicode 分词，对 CJK 字符的支持有限。若需要精确的东亚语言检索，请在 configs/default.yaml 中将 index.analyzer 字段设置为 "cjk" 并重建索引。重建命令为 go run cmd/indexer/main.go --rebuild-index。需注意 CJK 分析器会增大索引体积约 30%。
