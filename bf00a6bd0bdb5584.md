# CSU-TKG-Index

CSU-TKG-Index 是一个面向技术知识图谱构建与开源资源外链聚合的静态站点生成工具。项目定位于为开发者、技术研究员以及开源社区贡献者提供一套结构化的 Markdown 资源索引体系，能够将分散在代码仓库中的大量技术文档、教程笔记与配置参考通过统一的目录树与元数据标注进行归集和呈现。

本项目并非一个传统的动态 Web 应用，而是一套基于文件系统的文档索引方案。它通过解析特定仓库下的 Markdown 文件集合，提取文件头部的元信息与正文摘要，生成可供静态站点使用的 JSON 索引数据与导航目录。目标用户包括需要维护大型技术文档库的团队、希望建立个人知识管理体系的开发者，以及从事开源项目生态分析的研究人员。项目解决了技术资源分散、检索困难、缺乏上下文关联的痛点，通过标准化的文件命名与目录结构，使得数千个资源链接可以被系统化地管理和展示。

## 功能概览

**结构化资源索引**：扫描指定仓库目录下的所有 Markdown 文件，根据文件名哈希与特定前缀规则生成索引条目，每条记录包含文件路径、提交哈希摘要和分类标签。

**元数据提取与缓存**：从每个 Markdown 文件的内容中提取一级标题、创建时间、最后修改时间和自定义标签，生成 JSON 格式的元数据缓存文件，减少重复 IO 开销。

**静态导航页生成**：根据索引数据自动生成按字母顺序排列的资源导航页面，支持按文件名的十六进制哈希前缀进行分组浏览，适配大规模资源列表的展示需求。

**增量更新机制**：通过对比本地缓存与远程仓库的最新提交记录，仅处理新增或发生变更的 Markdown 文件，显著提升大型资源库的同步效率。

**链接有效性检查**：内置基础的 URL 格式校验与重复链接检测功能，帮助维护者及时发现资源列表中的失效或重复条目。

**多格式导出支持**：索引数据可导出为 JSON、CSV 和纯文本列表三种格式，便于下游工具链进行二次处理或导入其他知识管理平台。

## 应用场景

技术文档库的自动化索引生成。当一个开源项目积累了数百篇技术笔记或教程文档时，维护者可以使用本工具定期扫描文档目录，自动生成按文件名排序的索引页面，无需手动维护冗长的目录列表。

个人知识管理的资源归集。独立开发者或技术博主可以将自己在 GitHub 上收藏的各类技术文章链接统一存放在一个仓库中，通过本工具生成带有分类标签的导航页面，方便日后检索和回顾。

开源社区的学习路径整理。社区贡献者可以将一系列由浅入深的学习资源整理成 Markdown 文件集合，利用本工具的索引功能生成结构化的学习路径导航，降低新成员的入门门槛。

## 快速开始

以下命令演示了如何从 GitHub 克隆资源仓库、安装依赖并运行索引生成流程。

```bash
git clone https://github.com/warriorjacketenact/csutkg.git
cd csutkg
npm install
npm run build -- --input ./main --output ./dist
```

执行上述命令后，工具将扫描 ./main 目录下的所有 Markdown 文件，生成索引数据并输出到 ./dist 目录。如需监听文件变化并自动重新构建，可使用 npm run watch 命令。

## 安装要求

项目基于 Node.js 运行时构建，依赖以下环境与库。

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js >= 18.0.0 | 是 | 项目运行时环境，需支持 ES Module 语法 |
| npm >= 9.0.0 | 是 | 包管理器，用于安装项目依赖 |
| marked >= 4.3.0 | 是 | Markdown 解析器，用于提取文件内容中的标题与元数据 |
| glob >= 10.3.0 | 是 | 文件匹配库，用于递归扫描指定目录下的 .md 文件 |
| fs-extra >= 11.1.0 | 是 | 扩展的文件系统操作库，用于读写缓存和输出文件 |
| commander >= 11.0.0 | 是 | 命令行参数解析库，用于处理构建命令的输入选项 |
| chokidar >= 3.5.3 | 否 | 文件监听库，仅在开发模式或 watch 命令下使用 |
| eslint >= 8.0.0 | 否 | 代码检查工具，仅在代码贡献和测试环境中使用 |

## 项目结构

项目采用标准的 Node.js 应用目录布局，核心逻辑位于 src 目录下，缓存与输出目录在运行时自动生成。

```
csutkg/
├── src/                          # 源代码主目录
│   ├── core/                     # 核心处理模块
│   │   ├── scanner.js            # 文件扫描器，负责遍历目录并生成文件列表
│   │   ├── parser.js             # Markdown 解析器，提取标题和元数据
│   │   └── indexer.js            # 索引生成器，汇总数据并构建索引对象
│   ├── cli/                      # 命令行接口模块
│   │   ├── commands/             # 子命令实现
│   │   │   ├── build.js          # build 命令实现，执行完整构建流程
│   │   │   └── watch.js          # watch 命令实现，启动文件监听模式
│   │   └── main.js               # CLI 入口，注册命令和解析参数
│   ├── utils/                    # 通用工具函数
│   │   ├── hash.js               # 文件名哈希计算与校验工具
│   │   ├── cache.js              # 缓存读写与增量更新判断逻辑
│   │   └── validator.js          # URL 格式校验与重复链接检测
│   └── config/                   # 配置管理
│       └── defaults.js           # 默认配置项，包含输入输出路径及忽略规则
├── tests/                        # 单元测试与集成测试
│   ├── unit/                     # 各模块的单元测试用例
│   └── fixtures/                 # 测试用的模拟文件与样本数据
├── dist/                         # 构建输出目录（运行时自动生成）
├── cache/                        # 元数据缓存目录（运行时自动生成）
├── docs/                         # 项目文档
│   ├── guide/                    # 用户指南文档
│   ├── maintenance/              # 维护手册
│   └── reference/                # 参考文档

└── README.md                     # 项目说明文档
```

## 贡献指南

我们欢迎社区贡献者提交 Pull Request 或报告问题。请按照以下步骤进行协作。

第一步：Fork 本仓库并克隆到本地。在本地仓库中运行 npm install 安装所有开发依赖，确保使用 Node.js 18 及以上版本。

第二步：创建新的功能分支，分支名称应遵循 feat/xxx 或 fix/xxx 的格式。所有的代码修改应附带对应的单元测试，测试用例放置在 tests/unit 目录下，并确保现有测试全部通过。

第三步：提交代码时请遵循 Conventional Commits 规范，使用 feat:、fix:、docs:、chore: 等类型前缀。提交前运行 npm run lint 和 npm run test 进行代码检查和测试验证。

第四步：向本仓库的 main 分支发起 Pull Request，并在 PR 描述中清晰说明改动内容、关联的 Issue 编号以及测试覆盖情况。PR 需要至少一位维护者进行 Code Review 后方可合并。

## 常见问题

Q: 构建过程提示 "No markdown files found" 错误，如何解决？

A: 此错误表示工具在指定的输入目录下未找到任何 .md 文件。请检查 --input 参数是否指向了正确的目录路径，并确认该目录下确实存在扩展名为 .md 的文件。如果目录路径正确但仍报错，请检查目录的读取权限以及文件是否被 .gitignore 或其他忽略规则排除。

Q: 索引生成后，部分文件的标题未被正确提取，显示为默认文件名，是什么原因？

A: 工具使用 marked 库解析 Markdown 内容，仅提取文件中的第一个一级标题（# 标题）。如果文件开头没有使用 # 定义标题，或者标题位于代码块或 HTML 注释之后，则解析器可能无法识别。建议在每个 Markdown 文件的第一行使用 # 定义明确的标题，并确保标题前无其他非空白字符。

Q: 如何处理资源列表中大量链接的重复检测问题？

A: 工具在构建索引时会自动对提取到的所有链接进行去重检测，并将重复项记录在构建日志中。维护者可以通过启用 --verbose 参数查看详细的重复链接报告。如需手动清理重复条目，建议使用文本编辑器或脚本工具对 main 目录下的 .md 文件进行全局搜索和替换。
